# PRD: LLM-Powered News Article to 90-Second Video Generator

## 1. Product Summary

### Product Name

News-to-Video Generator (working title)

### One-line Description

A CLI-first, LLM-orchestrated system that converts a news article into a polished 90-second vertical or horizontal video using a structured storyboard pipeline, stock imagery, AI-generated images, voiceover, subtitles, motion graphics, and Blender-based rendering.

### Problem Statement

Creators, publishers, and social media teams spend significant time converting written news into short-form video. Manual production requires scripting, asset search, editing, timing, narration, and rendering. Existing tools are either too manual, too rigid, or too focused on generic AI video generation without editorial control. The product should automate the pipeline while still allowing strong narrative control and deterministic output.

### Product Vision

Transform a news article into a broadcast-quality short video with minimal human input by using an LLM as the planning layer and Blender as the rendering/compositing engine, with stock images and Stable Diffusion as asset sources.

### Target Output

* 90-second video
* Vertical 9:16 as default, with optional 16:9 export
* Voiceover
* On-screen captions
* Scene-based visuals
* Intro and outro branding
* Optional lower-thirds and headline cards

---

## 2. Goals and Non-Goals

### Goals

1. Convert a single news article into a coherent 90-second video.
2. Use an LLM to parse the article, generate a storyboard, and select visual strategy per scene.
3. Support a hybrid visual pipeline using:

   * stock images for real-world events and factual scenes
   * Stable Diffusion for abstract, symbolic, or missing visuals
   * Blender for compositing, motion graphics, and final rendering
4. Produce output that is repeatable, configurable, and auditable.
5. Enable a CLI-based or backend API-based workflow.
6. Allow template-driven editing rather than free-form prompt-to-video chaos.

### Non-Goals (MVP)

1. Real-time live editing.
2. Full non-linear editor timeline comparable to Premiere Pro.
3. User-authored hand-drawn keyframe animation.
4. Human-level editorial judgment without any structured constraints.
5. Long-form documentaries or multi-minute explainers.
6. Fully autonomous factual verification from the web in MVP.

## 6. User Experience Overview

### CLI Experience

User runs a command such as:

```bash
news2video --input article.txt --style breaking-news --aspect 9:16 --duration 90 --output out.mp4
```

### Expected UX Flow

1. User provides article or URL/text.
2. System extracts and normalizes article content.
3. LLM produces structured story plan.
4. Asset planner decides for each scene:

   * stock image
   * Stable Diffusion image
   * text-only motion graphic
   * quote card
   * chart/infographic
5. Voiceover script and subtitles are generated.
6. Assets are fetched or generated.
7. Blender composes scenes into a timeline.
8. Final render is produced and optionally post-processed by FFmpeg.
9. User receives video and a manifest file.

## 7. End-to-End Pipeline

### Stage 0: Ingestion

**Input types**

* Raw article text
* HTML page
* URL
* Markdown file
* RSS item
* Document file

**Responsibilities**

* Clean HTML
* Remove boilerplate
* Detect language
* Normalize punctuation and whitespace
* Store canonical article text

**Outputs**

* clean_article_text
* article_metadata
* source_url
* publish_date if available
* language

---

### Stage 1: Article Understanding

The LLM or preprocessing layer extracts:

* headline
* subhead
* article summary
* key entities
* events
* sentiment/tone
* topic category
* time sensitivity
* visualizable concepts
* quotes suitable for on-screen text
* factual highlights

**Output format** should be structured JSON, not free text.

Example output:

```json
{
  "headline": "...",
  "summary": "...",
  "tone": "serious",
  "topic": "politics",
  "key_points": [
    {"text": "...", "importance": 0.95},
    {"text": "...", "importance": 0.80}
  ],
  "entities": [
    {"name": "...", "type": "person"}
  ],
  "visual_hooks": ["crowd", "city skyline", "press conference"]
}
```

---

### Stage 2: Storyboard Generation

The LLM converts the article into a scene plan.

**Storyboard constraints**

* Total duration fixed at 90 seconds.
* Scenes should have explicit start/end timestamps.
* Each scene must have a purpose.
* Each scene must define narration text, on-screen text, and visual plan.
* The storyboard should allocate time proportionally to importance.

**Recommended scene model**

* Scene 1: Hook / opening headline
* Scene 2–N: Main facts and context
* Scene N-1: Summary / implications
* Scene N: Outro / call to action / brand close

**Storyboard schema example**

```json
{
  "duration_seconds": 90,
  "scenes": [
    {
      "id": "scene_01",
      "start": 0,
      "end": 8,
      "type": "hook",
      "narration": "...",
      "on_screen_text": "Breaking News",
      "visual_strategy": "stock_image",
      "visual_prompt": "...",
      "motion": ["fade_in", "slow_zoom"],
      "audio_cue": "intro_hit"
    }
  ]
}
```

---

### Stage 3: Visual Strategy Selection

Each scene should be assigned one visual strategy.

#### Strategy A: Stock Images

Use when the article refers to real-world places, public figures, institutions, events, or generally searchable concepts.

**Best for**

* cities
* buildings
* meetings
* maps
* public protests
* generic business settings
* natural disasters when plausible footage is unavailable

**Pros**

* realistic
* fast to source
* usually lower hallucination risk than AI-generated visuals

**Cons**

* may not match the exact event
* licensing must be managed carefully
* image search can fail on niche topics

#### Strategy B: Stable Diffusion Images

Use when the article contains abstract, symbolic, or hard-to-source visuals.

**Best for**

* dramatic concept shots
* symbolic representations
* futuristic visuals
* generic scenes missing available stock
* stylized editorial motion backgrounds

**Pros**

* flexible prompt control
* consistent visual style when seeded correctly
* local or self-hosted deployment possible

**Cons**

* factual accuracy risk if used for real-world people/events
* occasional visual artifacts
* style drift across multiple generations

#### Strategy C: Blender Motion Graphics

Use for

* headlines
* transitions
* lower thirds
* animated text
* charts
* map callouts
* layered compositing
* animated overlays

**Pros**

* deterministic and precise
* excellent for motion graphics
* supports timelines and keyframes

**Cons**

* more engineering overhead
* requires scene templates
* not ideal as a standalone image source

#### Hybrid Rule

The system should combine all three based on scene type, not treat them as competing replacements.

---

### Stage 4: Script Generation

The script generation layer produces:

* narration text
* subtitle text
* on-screen callouts
* scene transitions
* image prompts
* Blender template selection

**Important principle**
The LLM should not be allowed to output arbitrary Blender code in the main production path. It should output structured scene specs that map to prebuilt templates.

---

### Stage 5: Asset Acquisition and Generation

#### 5.1 Stock Image Retrieval

The system should query one or more stock sources.

**Selection logic**

* Use keywords and entities from the article.
* Prefer editorial or contextual images.
* Score candidates by relevance, clarity, aspect ratio, and license suitability.

**Fallbacks**

* If no suitable image exists, use AI-generated image.
* If neither exists, use motion graphic or text-based scene.

#### 5.2 Stable Diffusion Generation

The system should create a prompt from the scene spec.

**Prompt template inputs**

* scene topic
* tone
* color palette
* aspect ratio
* visual style
* negative prompt
* branding rules

**Prompt output example**

```text
A dramatic editorial illustration of a financial district during market turmoil, red downward arrows, cinematic lighting, modern news graphics style, high contrast, 9:16 composition
```

**Generation controls**

* seed management for consistency
* number of samples
* resolution presets
* negative prompts
* style presets per brand

#### 5.3 Asset Storage

All assets should be stored in a local cache and asset registry.

**Each asset record should include**

* asset_id
* source type
* source path or URL
* license or usage metadata
* prompt used for generation
* seed used
* generation timestamp
* scene_id linkage
* status

---

### Stage 6: Voiceover Generation

The system should generate a narration script and convert it into speech.

**Voiceover requirements**

* clear articulation
* neutral news tone by default
* support for multiple voices and languages
* sentence-level timing metadata if possible

**Important**
Voiceover should be generated before final compositing so that scene timing can align with spoken duration.

---

### Stage 7: Subtitle Generation

Subtitles should be derived from the narration script or from forced alignment.

**Subtitle responsibilities**

* word or sentence timing
* safe line wrapping
* readability on mobile devices
* accurate punctuation and casing

**Optional subtitle styles**

* broadcast style
* minimalist style
* punchy social style

---

### Stage 8: Blender Composition Layer

Blender is the render/compositing engine.

#### Responsibilities

* create scene templates
* load images and generated assets
* animate camera moves
* animate text and overlays
* compose clips into one timeline
* render final output

#### Recommended Blender usage model

Use a Python automation layer to generate and render scenes.

**Preferred approach**

* Maintain reusable template scenes in Blender.
* Generate a scene manifest JSON.
* Use a Python script to instantiate scenes, keyframes, image planes, text objects, audio tracks, and transitions.

#### Why Blender is useful here

* deterministic animation
* strong timeline control
* supports high-quality motion graphics
* works in headless mode
* can produce consistent brand templates

---

### Stage 9: Final Rendering and Post-Processing

After Blender renders the video, use a post-processing stage.

**Post-processing tasks**

* compress with FFmpeg
* normalize audio
* add loudness correction
* produce multiple aspect ratios if needed
* create thumbnails
* optionally burn subtitles or create separate subtitle files

## 10. Prompt and Schema Design

### Principle

Prompts should be constrained and evaluated against a strict schema.

### LLM Outputs Needed

1. Article summary JSON
2. Scene storyboard JSON
3. Visual prompt JSON
4. Voiceover script JSON
5. Brand style JSON
6. Safety/factual risk tags

### Important Constraints

* output must be valid JSON
* all scenes require timestamps
* all scene times must sum to approximately 90 seconds
* every scene must have one visual strategy
* no unsupported free-form animation instructions

---

## 11. Timing and Scene Allocation Rules

### Timing Policy

* total duration fixed to 90 seconds
* opening hook: 5–10 seconds
* body scenes: 60–70 seconds
* closing recap: 5–10 seconds

### Scene Duration Guidelines

* headline cards: 3–6 seconds
* quote scenes: 4–8 seconds
* chart scenes: 6–12 seconds
* stock image scenes: 5–8 seconds
* AI image scenes: 5–8 seconds

### Timing Engine Responsibilities

* compute target duration from script length
* adjust scene lengths to narration pace
* allow pauses for visual emphasis
* ensure no scene feels rushed or empty

### Brand Assets

* logo
* intro animation
* outro card
* font set
* color palette
* lower-third template
* caption style

### Default News Style

* clean sans-serif font

* subtle motion
* dark or neutral background
* red accent for breaking news
* blue accent for general news

### Motion Language

* fade in/out

### Composition Rules

---

## 13. Stock Image Subsystem Requirements

### Search Pipeline

* generate search keywords from article entities and events
* generate fallbacks and synonyms
* search by topic, location, object, and mood

### Ranking Criteria

* relevance to scene
* visual quality
* aspect ratio suitability
* freshness or editorial value
* license compatibility

### Retrieval Failures

If stock image search fails, the system should automatically switch to:

1. AI image generation
2. abstract motion background
3. text-card scene

### Asset Deduplication

Avoid repeating visually similar assets across adjacent scenes unless intentional.

---

## 14. Stable Diffusion Subsystem Requirements

### Model Hosting Options

* local GPU
* self-hosted server
* inference microservice

### Minimum Features

* text-to-image generation
* image-to-image optional
* negative prompts
* seed control
* batch generation
* style presets

### Prompt Engineering Requirements

Every scene prompt should include:

* subject
* composition
* style
* lighting
* mood
* aspect ratio
* negative constraints

### Quality Control

The system should reject images that:

* are blurry
* have obvious deformities in human subjects
* contain unreadable text
* conflict with the article meaning
* fail to match aspect ratio expectations

---

## 15. Blender Subsystem Requirements

### Blender Should Handle

* timeline assembly
* keyframe animation
* image scaling and cropping
* overlay placement
* text animation
* transitions
* audio track insertion
* render settings

### Template Library

Maintain predefined scene templates such as:

* headline card
* image + text split screen
* quote scene
* bullet list scene
* infographic scene
* closing card

### Blender Automation Strategy

Use Python scripts to:

* open a base project
* inject assets and metadata
* place objects by scene template
* assign keyframes
* render frames
* export final MP4

### Why Templates Matter

Templates keep output stable, reduce hallucination risk, and prevent the LLM from inventing unsupported layouts.

## 17. Error Handling and Fallback Logic

### Common Failures

* article text too short or too long
* LLM produces invalid JSON
* stock image API returns nothing
* Stable Diffusion generation fails
* voice synthesis error
* Blender render crash
* missing font or asset

### Fallback Rules

* regenerate once with adjusted prompt
* switch to simpler scene template
* replace missing visual with branded motion background
* shorten or merge adjacent scenes
* emit partial job report if final render fails

### Retry Policy

* bounded retries per stage
* stage-specific backoff
* idempotent asset generation when possible

---

## 18. Safety, Ethics, and Editorial Integrity

### Editorial Risks

* hallucinated facts
* misleading visuals
* incorrect attribution
* emotionally manipulative editing
* inappropriate use of AI-generated imagery for real events

### Safety Rules

* preserve factual meaning from source article
* avoid inventing people or events
* do not represent AI-generated images as real photos when the source is not real
* label synthetic visuals internally
* keep a provenance log

### Sensitive Content

For sensitive news categories such as violence, disasters, politics, or legal matters, require stricter factual checks and more conservative visuals.

## 26. Design Principles

1. Structure first, generation second.
2. The LLM plans; templates execute.
3. Use stock images when they are accurate.
4. Use Stable Diffusion when stock is unavailable or too abstract.
5. Use Blender for motion, layout, and final polish.
6. Keep every scene explainable and traceable.
7. Make failure recoverable.
8. Prefer deterministic output over creative freedom in the core pipeline.

## 30. Final Recommendation

Build the system as a **planner + renderer** architecture:

* The **LLM** is the planner and scene designer.
* The **asset layer** chooses between stock imagery and Stable Diffusion.
* **Blender** is the final deterministic scene compositor.
* **FFmpeg** handles final packaging and compression.

The core product is not just video generation; it is a structured editorial-to-video pipeline with reproducible scene logic, asset provenance, and motion template execution.
